When I last touched my side project in April 2025, I was riding high on Gemini 1.5 Flash's capabilities, architecting elaborate MCP integrations, and feeling pretty clever about my prompt engineering. Fast forward to October, and the landscape has shifted so much that my project feels like it was built in a different era.

## The Humbling Reality of AI Time Dilation

Here's what nobody tells you about building with AI: the half-life of your implementation decisions is measured in weeks, not years. That sophisticated MCP setup I spent weeks perfecting? It's now the equivalent of hand-cranking your car to start it. Sure, it works, but there are far better ways.

The models themselves have evolved from "pretty good at understanding papers" to "casually solving problems I didn't even know I had." Claude 3.5 Sonnet can now read documents better, understands pedagogical intent, identifies learning patterns, and generates insights that would make veteran teachers nod (I work on an edtech platform, so this matters).

Gemini 2.0 Pro's multimodal capabilities mean it can now understand handwritten mathematical proofs, sketched diagrams, and even the frustrated scribbles in margins that tell more about understanding than any final answer.

But none of this matters if you can't actually use it.

## The Great Context Management Migration

We all rushed to implement and integrate MCP because every company needed an "AI strategy." Well, it turns out that spending tens of thousands of tokens just to tell an AI how to use your tools is like hiring a PhD to operate a light switch.

Since then we have subagents, .md files, .cursorrules, implicit caching, custom tool calling, manual context managing, and now Claude Skills (... also just .md files afaik).

## The Agentic Revolution Nobody Prepared You For

Personally, I'm a bit skeptical about this whole agentic coding revolution. A lot of my friends do not write any code at all. Instead, they consider themselves the conductors of an orchestra of AI agents (and make me call them maestro getting angry at me when I use their first name), running 4-8 parallel terminals with different AI contexts. Back in April, this was impossible. We were all still using Cursor Composer and manually approving changes. Should I still be doing this, or do I have to go matrix mode?

## Agents All The Way Down

The developer workflow isn't the only place agents multiplied. A lot of projects rely on agents. What used to be deterministic logic became monolithic prompt chains which became specialized agent roles.
Each with isolated context, each composable, each orchestrated by a lightweight coordinator that routes tasks to the right specialist. 
Six months ago, this would've been a single 10,000-token prompt with conditional logic. Today, it's several focused agents, each expert in its domain.

## The Paradox of Model Selection

Here's something counterintuitive: newer isn't always better. That sparkling new Claude 3.5 Sonnet might actually perform worse on your specific task than an older version you dialed in six months ago.

Why? Because models aren't just getting "better"—they're getting different. Optimized for different benchmarks, trained on different data, aligned with different values. That aggressive, slightly unhinged Claude that would happily attempt any task? Replaced by a more thoughtful, cautious sibling that might politely decline to parse documents it deems "too ambiguous."

This is why the single most important addition to any AI project isn't a new feature—it's an evaluation suite.

## Building for AI Erosion: The Evaluation Imperative

Let's get into the bulk of this.

If I could send one message back to myself six months ago, it would be: "Build evals first, features second."

Here's the problem with coming back to an AI project after months away: you forgot what "good" looks like. That agent that felt adequate for your OCR task? Was it actually 90% accurate or 60% accurate? You have no idea.

Now let's say Gemini 3.0 Pro just dropped. Should you upgrade? The marketing says it's better at OCR, better at reasoning, better at everything. But better for YOUR use case?

Without evals: You spend 2 weeks migrating, manually testing, discovering edge cases, second-guessing every decision. You ship it. Two weeks later, users report lower quality output. Was it the model? The prompts? Both? You're flying blind.

With evals: You run your test suite against Gemini 2.5, 2.0, and even 1.5. A few minutes later you have data:
- Accuracy: 87% → 91%
- OCR parsing success: 82% → 95%
- Feedback quality: 88% → 84%
- Cost per run: $0.12 → $0.18

An OCR evaluation suite typically includes:
- A corpus of documents/inputs with known good outputs
- Edge cases: barely legible handwriting, mixed languages, creative formats
- Alignment tests: does the output actually help the user?
- Performance benchmarks: speed, token usage, cost per document

With proper evals, model updates become a simple decision: run the new model, compare metrics, deploy if better. Without them, you're making decisions based on pure vibes and marketing copy.

Evals don't just test your system they ARE your documentation. They encode what "good" means when you've forgotten. They turn gut-feel decisions into data-driven ones.

Three months from now, when you've forgotten everything again, your eval suite remembers.

Unfortunately, building an eval suite opens a whole new can of worms. The eval landscape is maturing just as fast as the models. Choosing a platform is a project in itself.

## The Art of Strategic Procrastination

Looking at my TODO.txt file is like reading archaeological layers of ambition. "Implement plagiarism checker"—sure, but now models can detect plagiarism as a side effect of understanding. "Email user results"—or maybe we should wait for the inevitable AI-native communication paradigm.

Here's what I almost did: implement Claude Skills because it seemed important, adopt LangGraph because it looked modern, normalize my database because it felt "right."

Here's what I did instead: Research first.

Skills Investigation:

Researched actual Claude Skills docs

Realized it's just YAML frontmatter + markdown files

My current prompts.py already solves this problem

Would add file management overhead without solving actual pain

Conclusion: Not today

Framework Investigation:

Researched LangGraph, CrewAI, AutoGen

All designed for multi-agent orchestration

My use case: single agent, multiple models

Conclusion: Keep it simple. (More on this in a future post).

Database Normalization Investigation:

Yes, it's good architecture

No, it doesn't solve current pain

Current schema works fine

Without evals, can't measure if it improves anything

Conclusion: Defer until after evals exist

The framework that matters: Research → Measure → Decide

Never implement without data. Never adopt without understanding your actual problem first. Never choose interesting over boring without a very good reason.

Practical Strategies for Project Resurrection
1. Archaeology Before Architecture
Before touching any code:

Read your old TODOs and laugh/cry

Run the existing system and marvel that it still works

Document what problems you were actually solving (they're easy to forget)

2. Resist the Rewrite Urge
Yes, everything could be better with the new models. No, you shouldn't rewrite everything. Pick one core flow—for my project, it was the OCR-to-JSON pipeline—and modernize that first.

3. Context Is Everything
Modern AI development is 90% context management. Whether it's Claude Skills, Cursor Rules, or just well-organized Markdown files, invest in making your project's context AI-legible. Your future self (in 3 months) will thank you.

4. Embrace Ephemeral Expertise
That deep knowledge of MCP internals? Let it go. That perfect prompt you crafted? It's already obsolete. Focus on principles, not implementations. Understand why things work, not just how.

5. Build for Obsolescence
Every line of code you write should assume the underlying models will change dramatically in 3 months. Abstractions aren't just good practice—they're survival strategies.

The Uncomfortable Truth About AI Velocity
Here's what keeps me up at night: we're all building on quicksand. Not technically—the APIs are stable, the models are reliable—but conceptually. The fundamental assumptions about what AI can and can't do shift monthly.

That OCR system that seemed impossibly ambitious in April? It's now table stakes. The real innovation edge has moved to predictive analytics, user style adaptation, and personalized output generation. By the time I implement those, the frontier will have moved again.

Conclusion: Surfing the Exponential
Returning to an AI project after six months isn't really about updating dependencies or integrating new models. It's about accepting that you're not the same developer, working with the same tools, solving the same problems.

The project that emerges from this revival won't be version 2.0—it'll be something fundamentally different, built with tools and techniques that didn't exist when I started. And in six months, when I inevitably return to it again, I'll go through this entire process once more.

The key isn't to resist this cycle—it's to design for it. Build systems that expect to be obsolete. Write documentation that assumes the reader (future you) has forgotten everything. Create abstractions that can wholesale swap their implementations.

Most importantly, maintain humility. That clever solution you're implementing today? In six months, an AI will do it better with a three-word prompt.
